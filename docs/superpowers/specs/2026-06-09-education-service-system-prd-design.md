# 教育服务系统 PRD 设计

## 1. 产品定位与范围

本项目是一套面向教育/留学服务机构的内部与外部一体化智能服务系统，主要服务四类用户：访客、员工、学生、管理员。系统第一版不依赖现成 CRM、教务或申请进度系统，先在本项目内自建数据库、管理后台和业务接口。

系统采用 Dify + 自研系统的混合架构。Dify 负责 Agent 编排、意图识别、RAG 问答、客户资料理解、自然语言业务操作解析、报告草稿生成等 AI 能力。自研系统负责前端页面、数据库、权限、审批、工单、活动报名、报表展示、接口调用、操作日志和人工确认。

PRD 按业务模块拆为五大块：

- 客户研判
- 客服 Agent
- 企业智能助手
- 学生智能助手
- 智能报告

本 PRD 面向开发团队，重点提供模块边界、页面清单、数据表、接口、Dify 工作流、验收标准和任务矩阵。团队按业务模块分工开发。公共能力不单独成模块，而是放入各业务模块内部说明；跨模块统一角色、状态枚举、接口风格、日志字段和 AI 草稿确认规则。

## 2. 架构与模块边界

系统分为四层：前端入口、自研后端、Dify 工作流、数据与知识库。

前端入口包括访客客服入口、员工端、学生端和管理后台。自研后端是业务事实源，负责账号权限、业务 CRUD、草稿确认、状态流转、操作日志和报表数据聚合。Dify 负责 AI 编排，包括意图识别、RAG 检索、客户资料解析、自然语言操作解析和报告草稿生成。数据库承载结构化业务数据，知识库承载公司资料、业务资料、留学政策、FAQ、海外生活指南、升学项目资料等非结构化内容。

核心边界：

- Dify 不直接写正式业务结果。
- 所有 AI 输出先生成草稿，人工确认后才能生效。
- 自研后端负责权限校验、事务写入、状态流转、日志记录和异常处理。
- 按业务模块交付，每个模块独立定义页面、表、接口、工作流和验收标准。
- 跨模块统一角色粗粒度：管理员、员工、学生、访客。

## 3. 客户研判模块

### 3.1 目标与用户

客户研判模块面向员工和管理员。员工上传客户资料，系统调用 Dify 工作流解析客户信息，并基于《用户画像研判规则》判断是否符合目标客户画像。第一版只输出规则判断结果，不自动创建意向客户，也不触发销售跟进流程。

### 3.2 页面

- 客户研判上传页
- 研判记录列表页
- 研判详情页

### 3.3 核心流程

1. 员工上传客户资料，支持文本、PDF、Excel。
2. 系统保存附件并调用 Dify 解析资料。
3. Dify 输出客户特征、匹配结论和命中依据草稿。
4. 员工确认后，系统归档正式研判记录。
5. 员工或管理员可查询历史记录和详情。

### 3.4 数据表

- 客户研判记录表：记录上传人、结论、命中依据、草稿状态、确认人、确认时间。
- 资料附件表：记录文件名、文件类型、存储路径、解析状态。
- 客户特征快照表：记录 Dify 从资料中抽取出的关键特征。

### 3.5 接口

- 上传客户资料
- 生成研判草稿
- 确认研判结果
- 查询研判记录
- 查看研判详情

### 3.6 Dify / AI 能力

- 客户资料解析
- 关键特征抽取
- 用户画像规则匹配
- 规则命中依据生成

### 3.7 异常处理

- 文件解析失败：返回解析失败状态，允许重新上传。
- 资料信息不足：输出信息不足提示，不给出强判断。
- 规则文档缺失：禁止生成正式草稿，提示管理员补齐规则。
- AI 输出无法结构化：标记为生成失败，允许重新生成。

### 3.8 验收标准

- 能上传文本、PDF、Excel 三类资料。
- 能生成匹配或不匹配结论和命中依据。
- 能人工确认研判草稿。
- 能在列表中查询历史记录。
- 文件或 AI 失败时有明确错误提示。

## 4. 客服 Agent 模块

### 4.1 目标与用户

客服 Agent 面向访客，作为官网、小程序或嵌入式聊天入口。第一版覆盖公司信息咨询、公司业务查询、留学政策查询、课程与项目推荐、活动与讲座报名、FAQ 自助解答、日常闲聊。

### 4.2 页面

- 访客聊天入口
- 客服会话记录页
- 知识库资料管理页
- 课程项目管理页
- 活动管理页
- 活动报名列表页
- 客服回复草稿确认页

### 4.3 核心流程

咨询问答流程：

1. 访客发送问题。
2. Dify 识别意图并检索知识库。
3. Dify 生成回复草稿。
4. 员工确认后，对访客回复。
5. 系统记录会话和引用来源。

活动报名流程：

1. 访客询问活动或讲座。
2. 系统查询活动表并返回活动信息草稿。
3. 访客提供报名信息。
4. Dify 抽取报名字段并生成报名草稿。
5. 员工确认后写入活动报名表。

第一版支持对话内收集报名信息并写入本系统活动报名表。活动改期、取消、提醒通知作为后续增强。

### 4.4 数据表

- 客服会话表
- 客服消息表
- 知识库资料表
- FAQ 表
- 课程项目表
- 活动表
- 活动报名表
- 客服回复草稿表

### 4.5 接口

- 发起会话
- 发送消息
- 生成客服回复草稿
- 确认并发送回复
- 查询课程项目
- 查询活动
- 创建活动报名草稿
- 确认活动报名
- 查询会话记录

### 4.6 Dify / AI 能力

- 多意图识别
- RAG 知识库问答
- 课程和项目推荐话术生成
- 日常闲聊回复生成
- 活动报名信息抽取

### 4.7 异常处理

- 知识库无命中：提示无法确认，并建议转人工。
- 报名信息缺失：继续追问缺失字段。
- 活动已满或已结束：禁止生成报名确认。
- AI 回复风险较高：进入人工复核，不对外回复。

### 4.8 验收标准

- 访客能发起咨询并生成基于知识库的回复草稿。
- 员工能确认客服回复。
- 访客能在对话中提交活动报名信息。
- 员工确认后，报名记录进入活动报名表。
- 后台能查看会话记录和报名记录。

## 5. 企业智能助手模块

### 5.1 目标与用户

企业智能助手面向员工和管理员，是内部业务查询和操作入口。第一版覆盖意向客户录入与查询、客户状态更新、口述日报、管理日报查阅、组织架构查询、新人入职指引、学生业务综管。

学生业务综管包括成绩录入与查询、请假申请查看与审批、投诉反馈跟进处理。

### 5.2 页面

- 企业助手聊天页
- 意向客户管理页
- 日报管理页
- 学生成绩页
- 投诉反馈页
- 请假审批页
- 组织架构页
- 新人指南知识库页
- 操作草稿确认页
- 操作日志页

### 5.3 核心流程

自然语言查询流程：

1. 员工输入查询指令。
2. Dify 解析查询意图和条件。
3. 自研后端校验权限并查询数据。
4. Dify 或后端生成结果摘要草稿。
5. 员工确认查看结果。

自然语言写操作流程：

1. 员工输入新增、修改或删除指令。
2. Dify 解析操作对象、字段和值。
3. 自研后端校验权限并生成待确认操作草稿。
4. 员工确认后，系统执行正式写入。
5. 删除、批量修改、关键状态变更需要额外二次确认。
6. 系统记录操作日志和执行结果。

### 5.4 数据表

- 意向客户表
- 员工日报表
- 投诉反馈表
- 学生成绩表
- 请假申请表
- 组织架构表
- 新人指南知识库表
- 操作草稿表
- 操作日志表

### 5.5 接口

- 企业助手消息接口
- 意向客户 CRUD
- 员工日报 CRUD
- 投诉反馈 CRUD
- 学生成绩 CRUD
- 请假审批接口
- 组织架构查询接口
- 操作草稿生成接口
- 操作草稿确认执行接口
- 操作日志查询接口

### 5.6 Dify / AI 能力

- NL2SQL 查询
- 自然语言业务操作解析
- 口述日报结构化
- 日报摘要生成
- 新人指南 RAG
- 组织架构问答

### 5.7 异常处理

- 权限不足：禁止生成可执行操作，只返回权限提示。
- 操作对象不明确：继续追问补充信息。
- 解析结果低置信度：只生成草稿，不允许确认执行。
- 删除、批量修改、关键状态变更：必须二次确认。
- 执行失败：保留失败日志，返回失败原因。

### 5.8 验收标准

- 员工能通过自然语言查询业务数据。
- 员工能生成新增、修改、删除草稿。
- 确认后能执行正式写入。
- 高风险操作有二次确认。
- 所有操作可在日志中追溯。

## 6. 学生智能助手模块

### 6.1 目标与用户

学生智能助手面向学生，作为学生端自助服务入口。第一版覆盖行政服务、心理关怀、售后反馈、学业考务、进度追踪、生活支持和增值转化。

由于当前没有外部系统对接，考务、申请进度、签证进度等数据先在本系统内维护。

### 6.2 页面

- 学生助手聊天页
- 请假申请页
- 请假记录页
- 投诉建议页
- 进度查询页
- 考务查询页
- 生活支持问答页
- 增值项目推荐页

### 6.3 核心流程

请假流程：

1. 学生提交请假申请。
2. 系统写入请假申请表。
3. 员工端手动审批。
4. 学生查看审批结果。

投诉反馈流程：

1. 学生提交投诉或建议。
2. Dify 提炼工单摘要草稿。
3. 人工确认后写入正式投诉工单。
4. 员工在企业端跟进处理。
5. 学生查看处理状态。

心理关怀流程：

1. 学生进行情绪闲聊或情绪反馈。
2. Dify 识别情绪标签和分值。
3. 系统生成待确认情绪记录。
4. 确认后写入情绪记录表。

第一版心理关怀只做情绪识别和记录，不主动预警、不通知老师、不生成干预任务。

### 6.4 数据表

- 请假申请表
- 投诉反馈工单表
- 心理/情绪记录表
- 学生成绩表
- 考务事项表
- 申请进度表
- 生活知识库表
- 升学项目资料表

### 6.5 接口

- 学生助手消息接口
- 请假申请提交接口
- 请假结果查询接口
- 投诉建议提交接口
- 投诉状态查询接口
- 学生成绩查询接口
- 考务事项查询接口
- 申请进度查询接口
- 生活知识问答接口

### 6.6 Dify / AI 能力

- 学生多意图识别
- 投诉内容摘要
- 情绪标签和分值识别
- 生活知识 RAG
- 增值项目推荐话术生成

### 6.7 异常处理

- 请假信息缺失：提示学生补充时间、原因等字段。
- 工单摘要不完整：进入人工补充。
- 情绪识别低置信度：记录为待确认，不写入正式画像。
- 查询无数据：提示暂无记录，不生成虚假结果。

### 6.8 验收标准

- 学生能提交请假并查看审批结果。
- 学生能提交投诉并生成工单草稿。
- 学生能查询成绩、考务和申请进度。
- 学生能获得生活知识问答。
- 情绪识别只记录不预警。
- 所有 AI 结果先草稿确认。

## 7. 智能报告模块

### 7.1 目标与用户

智能报告面向管理员和员工。第一版覆盖五类报告：

- 全域客户经营分析报告
- 员工日报汇总报告（日）
- 员工日报汇总报告（周）
- 学生心理健康周报
- 投诉处理周报

报告统一采用数据聚合、AI 生成草稿、人工确认发布或导出的流程。

### 7.2 页面

- 报告中心
- 报告生成配置页
- 报告草稿编辑页
- 报告详情页
- 报告版本记录页

### 7.3 核心流程

1. 用户选择报告类型和统计周期。
2. 自研后端聚合对应业务数据。
3. Dify 根据数据生成报告草稿。
4. 用户编辑并确认草稿。
5. 系统生成正式报告版本。
6. 用户查看、导出或回溯历史版本。

### 7.4 报告数据来源

- 全域客户经营分析报告：客户研判记录、意向客户、活动报名、客户状态变化。
- 员工日报汇总报告（日）：员工日报表。
- 员工日报汇总报告（周）：员工日报表。
- 学生心理健康周报：心理/情绪记录表。
- 投诉处理周报：投诉反馈工单表。

学生心理健康周报只做整体趋势和标签统计，不做高危预警或干预任务。

### 7.5 数据表

- 报告任务表
- 报告草稿表
- 报告版本表
- 报告导出记录表

### 7.6 接口

- 创建报告任务
- 聚合报告数据
- 生成报告草稿
- 编辑保存报告草稿
- 确认发布报告
- 导出报告
- 查询历史版本

### 7.7 Dify / AI 能力

- 客户经营洞察生成
- 日报摘要与风险提炼
- 心理趋势摘要
- 投诉归因与处理建议
- 报告自然语言生成

### 7.8 异常处理

- 数据不足：生成数据不足提示，不输出强结论。
- AI 草稿生成失败：允许重新生成。
- 草稿未确认：不能发布正式版本。
- 导出失败：保留失败记录并允许重试。

### 7.9 验收标准

- 能按日、周或手动选择周期生成五类报告草稿。
- 报告草稿可编辑。
- 确认后生成正式版本。
- 支持查看历史版本。
- 支持导出。
- AI 洞察保留数据来源和生成时间。

## 8. 全局规则

### 8.1 AI 草稿确认

所有 AI 输出默认先生成草稿，人工确认后生效。适用范围包括客服回复、客户研判结果、活动报名字段抽取、自然语言业务操作、情绪识别记录、投诉摘要、报告内容和推荐话术。

删除、批量修改、关键状态变更需要二次确认。关键状态包括签约、流失、投诉已解决、请假审批通过或拒绝等会显著影响业务记录的状态。

### 8.2 角色权限

第一版角色按粗粒度设计：

- 管理员：系统配置、知识库、报告、所有模块数据管理。
- 员工：客户、客服、企业助手、学生业务和报告相关操作。
- 学生：学生端自助服务和个人数据查询。
- 访客：客服 Agent 咨询和活动报名。

### 8.3 日志与审计

涉及正式写入、确认、删除、批量修改、报告发布、客服对外回复的操作都必须记录日志。日志至少包含操作人、操作对象、操作类型、操作前后数据摘要、AI 草稿 ID、确认时间、执行结果。

### 8.4 状态枚举

AI 草稿状态：待生成、生成中、待确认、已确认、已驳回、生成失败。

业务操作状态：草稿、待二次确认、已执行、执行失败、已撤销。

报告状态：待生成、草稿、已发布、导出中、导出失败。

工单状态：待确认、待处理、处理中、已解决、已关闭。

请假状态：待审批、已通过、已拒绝、已取消。

## 10. 端到端验收场景

- 访客咨询活动并提交报名信息，员工确认后写入活动报名表。
- 员工上传客户资料，系统生成客户研判草稿，员工确认后归档。
- 员工输入自然语言删除指令，系统生成操作草稿并要求二次确认，确认后执行并记录日志。
- 学生提交请假申请，员工端审批，学生端查看审批结果。
- 学生提交投诉，系统生成工单摘要草稿，员工确认后进入投诉工单。
- 管理员生成投诉处理周报草稿，编辑确认后发布正式版本并导出。
